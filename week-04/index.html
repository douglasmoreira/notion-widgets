Interface Segregation Principle (ISP)
No client should be forced to depend on methods it does not use.

What is ISP?

The Interface Segregation Principle states that a class should never be forced to implement an interface it doesn't fully use. Instead of one large, general-purpose interface, you should define multiple smaller, focused interfaces — each representing a specific capability or contract.

Robert Martin coined it as a response to "fat interfaces" — interfaces that bundle unrelated methods together, forcing implementors to provide empty or fake implementations for methods they don't need.

The Mental Model

Think of ISP like a job description. A focused job description says exactly what this role does — "Processes payments". A bloated one says "Processes payments, sends emails, generates reports, manages users, and handles infrastructure". The second kind forces every hire to pretend they're qualified for everything, even if they only needed one skill.

Why it matters at Senior / Architect level

At a senior level, ISP is not just about clean interfaces — it's about how you design module boundaries. Violating ISP creates rigid coupling: a change in one unrelated method forces recompilation or retesting of components that have nothing to do with that change.

In your current environment, ISP appears directly in Hexagonal Architecture: each port (interface) should represent exactly one capability, not a bundle of everything the outside world might want to do with your domain. A violation here bleeds across the boundary and defeats the isolation Hexagonal Architecture is designed to provide.

ISP vs SRP — the key distinction

Principle Applies to Core question
SRP Classes / modules Does this class do too many things?
ISP Interfaces / contracts Does this interface demand too many capabilities from its implementors?

They often appear together: a fat class usually has a fat interface. But fixing the interface (ISP) doesn't automatically fix the class (SRP), and vice versa.

Composing multiple capabilities when needed

ISP doesn't mean a class can't implement multiple interfaces. It means each interface stays focused. A class that genuinely does email + SMS can implement both without being forced into everything else.

FullChannelNotifier.java — implements only what it supports
public class FullChannelNotifier implements EmailSender, SmsSender {

    @Override
    public void sendEmail(String to, String subject, String body) {
        // email logic
    }

    @Override
    public void sendSms(String phone, String message) {
        // SMS logic
    }
}

ISP in Hexagonal Architecture — Port design

OrderPort.java — fat port (violation)
// Too broad — a single port that owns all order operations
public interface OrderPort {
    Order createOrder(OrderRequest request);
    void cancelOrder(String orderId);
    OrderStatus getOrderStatus(String orderId);
    List<Order> getOrderHistory(String customerId);
    void updateShippingAddress(String orderId, Address address);
    byte[] generateOrderPdf(String orderId);
}
Segregated ports — clean hexagonal boundary
public interface OrderCreationPort {
    Order createOrder(OrderRequest request);
}

public interface OrderQueryPort {
    OrderStatus getOrderStatus(String orderId);
    List<Order> getOrderHistory(String customerId);
}

public interface OrderManagementPort {
    void cancelOrder(String orderId);
    void updateShippingAddress(String orderId, Address address);
}

public interface OrderDocumentPort {
    byte[] generateOrderPdf(String orderId);
}
Each adapter (REST controller, Kafka consumer, batch job) now only depends on the port it actually uses. A Kafka consumer that cancels orders only depends on OrderManagementPort — it never sees query or document methods.

Traps and senior-level insights

1. Splitting too far is also wrong. ISP doesn't mean one method per interface. If methods are always used together and always by the same clients, they belong in the same interface. Over-segregation creates an explosion of tiny interfaces that are harder to navigate than a single focused one.
2. UnsupportedOperationException is a red flag, not a solution. If you see this in a production codebase, it almost always signals an ISP violation. The implementor was forced into an interface that doesn't fit. This is dangerous because the violation isn't visible at compile time — it only blows up at runtime.
3. ISP applies to more than interfaces. In modern Java, this also applies to abstract classes, generic base classes, and even Spring service beans that expose too many methods. If a bean has 15 public methods and different callers only use 2–3 each, you have an ISP smell worth decomposing.
4. ISP and testing. Fat interfaces make mocking painful. In Mockito, mocking a 12-method interface to test a class that only uses 2 of them creates noise and fragility. Segregated interfaces produce smaller, clearer test doubles. If your test setup feels verbose, check whether the interface is too fat.
5. The connection to Dependency Inversion (next week). ISP sets up DIP naturally. When you have focused interfaces, it's easy to inject exactly the capability you need and swap implementations. Fat interfaces make DIP harder because the injected dependency brings unwanted methods with it.

Check your understanding

1. In one sentence: what does the Interface Segregation Principle prohibit?

ISP prohibits forcing a class to implement methods it does not use — that is, defining interfaces that are broader than any single client's actual needs.

2. What runtime symptom is almost always a sign of an ISP violation in Java?

UnsupportedOperationException thrown from interface implementations. It means the implementor was forced into a contract it doesn't actually support.

3. How does ISP relate to Hexagonal Architecture specifically?

In Hexagonal Architecture, each port is an interface representing a boundary capability. A fat port that bundles multiple unrelated capabilities forces every adapter (REST, Kafka, batch) to depend on methods it doesn't use — breaking the isolation the architecture is meant to provide. ISP keeps ports focused, so each adapter depends only on the capability it actually needs.

4. ISP says "split interfaces". Does that mean every method should have its own interface? Why or why not?

No. Over-segregation creates as many problems as fat interfaces — it fragments cohesion and makes the codebase hard to navigate. The right granularity is determined by client need: methods that are always used together by the same clients belong in the same interface. Split only when different clients need different subsets.

5. How does ISP affect unit testing with Mockito?

Fat interfaces force you to mock methods the class under test never calls, adding noise and fragility. Segregated interfaces produce smaller mocks that only cover the actual dependency, making tests cleaner and easier to read.

6. What is the key conceptual difference between SRP and ISP?

SRP applies to classes and modules — it asks whether a class has too many responsibilities. ISP applies to interfaces and contracts — it asks whether an interface demands too many capabilities from its implementors. A class can violate SRP without violating ISP, and vice versa, though they often appear together.
