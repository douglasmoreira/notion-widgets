Q1 Study Roadmap — Block 2 — Week 6 of 13
KISS, DRY & YAGNI
Write code that does exactly what it needs to — nothing more, nothing less.
Welcome to Block 2 — Clean Code Principles. You have just finished five weeks on SOLID. This block is about the habits that surround those principles: how you name things, how much complexity you allow, and how aggressively you cut what is not needed. These are not rules — they are intuitions to build. By the end of Week 7 you should be applying them without thinking.
The Three Principles

KISS, DRY, and YAGNI are often listed together as if they were one idea. They are not. Each targets a different failure mode in how developers write code. Understanding them separately is what allows you to apply them together.

Principle Full Name The failure it targets The question it asks
KISS Keep It Simple, Stupid Accidental complexity — code that is harder to read than the problem it solves Can a developer who did not write this understand it in 30 seconds?
DRY Don't Repeat Yourself Knowledge duplication — the same business rule or logic living in more than one place If this rule changes, how many places do I have to update?
YAGNI You Aren't Gonna Need It Speculative generality — building for requirements that do not exist yet Does a current, real requirement justify this code existing?
KISS — Keep It Simple, Stupid

Complexity is not the same as sophistication. Complexity is what happens when you solve a simple problem with more moving parts than necessary — nested conditionals, clever one-liners, chains of lambdas, deep inheritance trees. Sophistication is solving a genuinely hard problem in the clearest way possible.

The KISS test is about the next person, not you. You already understand your own code when you write it. KISS asks whether someone who has never seen it can understand it without asking you questions. At a senior level this matters more than ever: your code will be read in code reviews, maintained by teammates, and debugged under pressure at 11pm by someone who is not you.

The complexity trap: Many developers equate complex code with skilled code. Stream chains with five intermediate operations, method references where a simple loop would do, ternary operators nested three deep — these signal effort, not quality. At a Tech Lead level, you will eventually need to reverse someone else's "clever" code. The skill is knowing when simple is actually the harder choice.
DRY — Don't Repeat Yourself

DRY is widely misunderstood as "no duplicate lines of code." That is not what it means. The original definition from The Pragmatic Programmer is: every piece of knowledge must have a single, authoritative representation in the system.

The word knowledge is what matters. Two methods that happen to contain the same three lines of code are not necessarily a DRY violation — they might represent different business concepts that happen to look the same today. Merging them into a shared utility would couple two unrelated concerns. That is worse than the duplication.

A real DRY violation is when the same business rule — for example, "an order is eligible for a discount if its total exceeds R$200" — is hardcoded in three different services. If that threshold changes to R$250, you have three places to update, three chances to miss one, three potential bugs in production.

DRY in Hexagonal Architecture: Your ports are the authoritative source of truth for input/output contracts. If you find yourself duplicating validation logic between an inbound adapter (REST controller) and an application service, that is a DRY violation — the rule should live in the domain, not in two adapters independently.
YAGNI — You Aren't Gonna Need It

YAGNI is an Extreme Programming principle that answers a very specific temptation: building features, abstractions, or extension points for future requirements that have not been asked for yet.

The reasoning goes: requirements change. The feature you build today for "what we might need in six months" will almost certainly not match what is actually needed in six months. You will have paid the cost of building and maintaining it, and then pay again to change it. The correct response to a hypothetical future need is to write code that is easy to change, not code that pre-emptively handles every scenario.

YAGNI in architecture: This principle applies beyond code. Adding Kafka to a system "because we might need async messaging later" when all current requirements are synchronous is a YAGNI violation at the infrastructure level. It creates operational cost, deployment complexity, and a learning curve — for requirements that may never materialize.
How the three interact

Applied together, these principles create a specific discipline:

Danger zone
  • DRY without YAGNI → over-abstraction for hypothetical reuse
  • KISS without DRY → copy-paste sprawl, knowledge split everywhere
  • YAGNI without KISS → minimal features but tangled, unreadable code
Balanced
  • Extract a shared abstraction only when duplication is real and the concept is stable
  • Build only what is asked; make the code easy to extend when the need arrives
  • Prefer readable code over clever code; simple structure over elegant machinery

All examples are from a fictional OrderProcessingService in an enterprise backend context.

KISS — Accidental Complexity
❌ KISS Violation — clever but unreadable
// What does this actually do?
public Map<String, List<Order>> groupOrders(List<Order> orders) {
    return orders.stream()
        .filter(o -> o.getStatus() != null)
        .collect(Collectors.groupingBy(
            o -> Optional.ofNullable(o.getCustomer())
                         .map(Customer::getRegion)
                         .orElse("UNKNOWN"),
            Collectors.filtering(
                o -> o.getTotal().compareTo(BigDecimal.ZERO) > 0,
                Collectors.toList()
            )
        ));
}
✅ KISS Applied — clear intent, same result
public Map<String, List<Order>> groupOrdersByRegion(List<Order> orders) {
    Map<String, List<Order>> result = new HashMap<>();

    for (Order order : orders) {
        if (order.getStatus() == null) continue;
        if (order.getTotal().compareTo(BigDecimal.ZERO) <= 0) continue;

        String region = resolveRegion(order);
        result.computeIfAbsent(region, k -> new ArrayList<>()).add(order);
    }

    return result;
}

private String resolveRegion(Order order) {
    if (order.getCustomer() == null) return "UNKNOWN";
    return order.getCustomer().getRegion();
}
The second version is longer in line count but shorter in cognitive cost. Every step is explicit. A new team member can follow it without decoding the stream API chain.
DRY — Knowledge Duplication
❌ DRY Violation — same business rule in two places
// In OrderService
public void applyDiscount(Order order) {
    if (order.getTotal().compareTo(new BigDecimal("200.00")) > 0) {
        order.applyDiscountPercent(10);
    }
}

// In InvoiceService (weeks later, copy-pasted by another developer)
public BigDecimal calculateFinalAmount(Order order) {
    BigDecimal total = order.getTotal();
    if (total.compareTo(new BigDecimal("200.00")) > 0) {
        total = total.multiply(new BigDecimal("0.90"));
    }
    return total;
}
✅ DRY Applied — the rule lives in one place
// Domain rule — single authoritative source
public class DiscountPolicy {

    private static final BigDecimal DISCOUNT_THRESHOLD = new BigDecimal("200.00");
    private static final int        DISCOUNT_PERCENT   = 10;

    public boolean isEligible(Order order) {
        return order.getTotal().compareTo(DISCOUNT_THRESHOLD) > 0;
    }

    public BigDecimal apply(BigDecimal amount) {
        BigDecimal factor = BigDecimal.ONE
            .subtract(new BigDecimal(DISCOUNT_PERCENT).divide(new BigDecimal("100")));
        return amount.multiply(factor);
    }
}

// Both services now delegate to DiscountPolicy — one place to change
public void applyDiscount(Order order) {
    if (discountPolicy.isEligible(order)) {
        order.applyDiscountPercent(10);
    }
}

public BigDecimal calculateFinalAmount(Order order) {
    BigDecimal total = order.getTotal();
    return discountPolicy.isEligible(order)
        ? discountPolicy.apply(total)
        : total;
}
YAGNI — Speculative Generality
❌ YAGNI Violation — built for requirements that do not exist
// Current requirement: process credit card payments only.
// Developer decided to "future-proof" the service.

public class PaymentService {

    public void processCreditCard(PaymentRequest request) { /* real code */ }

    // Nobody asked for these. They are not in any ticket. They are not tested.
    public void processDebitCard(PaymentRequest request) {
        throw new UnsupportedOperationException("Not implemented yet");
    }

    public void processBoleto(PaymentRequest request) {
        throw new UnsupportedOperationException("Not implemented yet");
    }

    public void processPix(PaymentRequest request) {
        throw new UnsupportedOperationException("Not implemented yet");
    }

    public void processCrypto(PaymentRequest request) {
        throw new UnsupportedOperationException("Not implemented yet");
    }

    public void schedulePaymentForFutureDate(PaymentRequest request, LocalDate date) {
        throw new UnsupportedOperationException("Not implemented yet");
    }
}
✅ YAGNI Applied — only what is needed today
// Clean, focused, honest about what it does.
// When Pix is required, we add it then — with a real requirement,
// real tests, and full understanding of the integration.

public class PaymentService {

    public void processCreditCard(PaymentRequest request) {
        // validate, charge, persist, emit event
    }
}
YAGNI does not mean "write inflexible code." It means: write code that is easy to extend when the real need arrives, instead of pre-building the extension for a need that may never arrive.

These are the distinctions that separate a developer who knows these principles from one who actually applies them correctly under pressure.

1. DRY is about knowledge, not lines of code
common mistake

Two methods that contain the same three lines are not automatically a DRY violation. If those three lines represent two different business concepts that happen to look identical today, merging them creates a wrong abstraction. Example: validateUserAge() and validateDriverLicenseAge() both check age >= 18. They should stay separate — they represent different rules that will diverge the moment the legal ages change independently.
2. The wrong abstraction is worse than duplication
serious trap

Sandi Metz coined this: "duplication is far cheaper than the wrong abstraction." When you extract a shared method or class too early — before you truly understand what varies and what is stable — you couple unrelated concepts. Later, when they diverge, you have to either keep adding conditionals to the shared code or break it apart at high cost. Prefer duplication that is honest over abstraction that lies about what the code represents.
3. YAGNI applies at every level of the stack
Most developers think of YAGNI as a code-level principle. It applies equally to architecture. Examples of YAGNI at architecture level:
  • Adding Kafka when all current flows are synchronous and the team of 3 devs has no event-driven requirements
  • Building a multi-tenant system from day one for a product with a single client
  • Designing a plugin system before knowing what plugins will be needed
  • Implementing CQRS before there is a read/write performance problem that justifies it
4. KISS does not mean "no abstraction"
KISS is not an argument against well-named methods, clean interfaces, or separation of concerns. A service that delegates to a DiscountPolicy and an OrderValidator is simpler than one that does everything inline — because the intent is clearer. KISS targets unnecessary complexity, not appropriate structure. The question is always: does this complexity exist because the problem demands it, or because the developer was being clever?
5. These principles interact with SOLID
KISS supports SRP: a class with a single responsibility is simpler to understand than one that does many things.

DRY supports OCP: when knowledge lives in one place, extending behaviour means adding code, not hunting for duplicates to update.

YAGNI directly opposes premature application of design patterns: you do not need an Abstract Factory if you have one concrete implementation. Add the pattern when you have two or more concrete cases that justify it.

Five questions. Answer each before revealing. If you cannot answer confidently, go back to the Concept or Important Extras tab.

1. A colleague extracted a base class to avoid duplicating three lines of validation logic shared between CreateOrderHandler and UpdateOrderHandler. Six months later, the create rule changed and they had to add a boolean flag to the base class to control behaviour. What went wrong and which principle applies?

This is the wrong abstraction problem — a DRY violation applied prematurely. The two handlers shared identical code at the time but represented different business rules. When create logic diverged, the shared abstraction started lying about what the code did. The fix is to inline the logic back into each handler and accept the duplication, or create two distinct policy objects with a common interface only if the concepts are genuinely related. The principle is DRY, but the failure is caused by extracting the abstraction before understanding whether the duplication is accidental or structural.

2. What is the difference between a KISS violation and a YAGNI violation? Give a one-sentence definition of each.

KISS violation: code that is more complex than the problem it solves — unnecessary nesting, clever one-liners, over-engineered structure that makes it harder to read than a straightforward solution would be.

YAGNI violation: code that exists for a requirement that has not been asked for yet — features, extension points, or abstractions built speculatively for a hypothetical future that may never arrive.

3. You are reviewing a PR. The developer added a NotificationStrategy interface with three implementations (EmailStrategy, SmsStrategy, PushStrategy) — but the current system only ever sends email. Is this a YAGNI violation, a good design decision, or does it depend? Explain.

It depends — but it is most likely a YAGNI violation unless there is strong evidence that multiple channels are coming soon. The Strategy pattern is a valid choice when you have multiple concrete implementations and need to swap them at runtime. With only one implementation today, the interface adds indirection without benefit. However, if SMS and Push are confirmed requirements in the next sprint, the abstraction is justified — it avoids a larger refactor later. The rule of thumb: add the abstraction when you have the second concrete case, not before.

4. In your current Hexagonal Architecture codebase, where is the most likely place to find a DRY violation? Why?

The most common place is between inbound adapters and application services. A REST controller that validates a request before calling the service, and a service that re-validates the same input independently, is duplicating the validation rule. In Hexagonal Architecture, the application service (use case) is the authoritative owner of business rules. Input validation that is about the domain — not about HTTP specifics — should live there, not in both layers. Another common location is when multiple use cases independently implement the same eligibility check instead of delegating to a shared domain policy.

5. "YAGNI means you should never design for extensibility." Is this statement true or false? Explain why.

False. YAGNI means you should not build specific features or extension points for requirements that do not exist yet. It does not mean writing rigid, inflexible code. The correct interpretation is: write code that is easy to change, and change it when the real requirement arrives. Good practices like meaningful naming, small methods, and single responsibility make code easy to extend without pre-building the extensions. The distinction is: easy to extend (good) versus already extended for imagined futures (YAGNI violation).
