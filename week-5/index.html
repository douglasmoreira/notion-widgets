Dependency Inversion Principle (DIP)
High-level modules must not depend on low-level modules. Both should depend on abstractions.
What is DIP?

DIP is the "D" in SOLID and it is the most architecturally impactful of all five principles. It is the theoretical foundation of Hexagonal Architecture, the reason Spring's dependency injection exists, and the reason you can unit-test business logic without spinning up a database or hitting an HTTP endpoint.

Rule 1: High-level modules (domain, business logic, use cases) must NOT depend on low-level modules (databases, HTTP clients, messaging, file systems).

Rule 2: Both high-level and low-level modules must depend on abstractions (interfaces or abstract classes). Abstractions must not depend on details. Details must depend on abstractions.
DIP vs Dependency Injection — Critical Distinction

These are related but fundamentally different things. Confusing them is a senior-level trap.

DIP Dependency Injection (DI)
A design principle — tells you what should depend on what A pattern/technique — one way to achieve DIP
Architectural decision Implementation mechanism
You can apply DIP without a framework Spring's @Autowired and constructor injection are DI mechanisms
About direction of dependencies About how dependencies are supplied
You can use constructor injection and still violate DIP — if you inject a concrete class instead of an interface. DI without an abstraction is just fancy object creation, not DIP.
DIP in Your Current Stack
Your StackDIP in Action
Hexagonal Architecture Ports are the abstractions (interfaces). Adapters are the details (implementations). Domain never imports adapters directly.
Spring @Service / @Repository Spring injects the concrete implementation, but your service depends on the interface — not on the class.
Kafka consumers Domain defines an EventPublisher interface. A Kafka adapter implements it. Domain never imports the Kafka client library.
MongoDB repositories Domain uses a PaymentRepository interface. MongoDB provides the implementation behind it.
Mockito in unit tests You can mock any dependency because the high-level class depends on an interface, not a concrete class.
Why This Matters at Senior / Architect Level

Without DIP, your business logic is coupled to infrastructure. Swap MySQL for MongoDB — domain breaks. Swap REST for Kafka — domain breaks. With DIP, you replace the implementation behind an interface and the domain is completely untouched. This is exactly what Hexagonal Architecture formalises at scale.

Every time a senior engineer insists on programming to interfaces, or a tech lead pushes back on importing infrastructure classes into domain code — they are applying DIP. Understanding this principle lets you participate in those decisions, not just follow them.
❌ Violation — High-level class depends on concrete low-level class
// Low-level: concrete implementation
public class SmtpEmailSender {
    public void send(String to, String subject, String body) {
        System.out.println("Sending SMTP email to " + to);
    }
}

// High-level: OrderService hardwired to SmtpEmailSender
public class OrderService {
    private final SmtpEmailSender emailSender; // ← concrete class, not interface

    public OrderService() {
        this.emailSender = new SmtpEmailSender(); // ← hardwired, cannot substitute
    }

    public void placeOrder(Order order) {
        // business logic...
        emailSender.send(order.getCustomerEmail(), "Order confirmed", "Your order is placed.");
    }
}
If you switch from SMTP to SendGrid, you must modify OrderService — a high-level business class — to fix an infrastructure concern. You also cannot test OrderService without sending real emails.
✅ DIP Applied — Both depend on the abstraction
// The abstraction (Port in Hexagonal Architecture terms)
public interface NotificationSender {
    void send(String to, String subject, String body);
}

// Low-level detail — SMTP adapter
public class SmtpNotificationSender implements NotificationSender {
    @Override
    public void send(String to, String subject, String body) {
        System.out.println("[SMTP] Sending to " + to + " | " + subject);
    }
}

// Low-level detail — SendGrid adapter (swappable without touching domain)
public class SendGridNotificationSender implements NotificationSender {
    @Override
    public void send(String to, String subject, String body) {
        System.out.println("[SendGrid] Sending to " + to + " | " + subject);
    }
}

// High-level module — depends only on the interface
@Service
public class OrderService {
    private final NotificationSender notificationSender; // ← interface

    public OrderService(NotificationSender notificationSender) { // ← constructor injection
        this.notificationSender = notificationSender;
    }

    public void placeOrder(Order order) {
        // business logic...
        notificationSender.send(
            order.getCustomerEmail(),
            "Order confirmed",
            "Your order #" + order.getId() + " has been placed."
        );
    }
}
✅ Unit Test — only possible because of DIP
class OrderServiceTest {

    @Test
    void shouldSendNotificationWhenOrderPlaced() {
        // Arrange — mock the interface, not a concrete class
        NotificationSender mockSender = Mockito.mock(NotificationSender.class);
        OrderService service = new OrderService(mockSender);
        Order order = new Order("42", "douglas@example.com");

        // Act
        service.placeOrder(order);

        // Assert
        Mockito.verify(mockSender).send(
            eq("douglas@example.com"),
            eq("Order confirmed"),
            contains("#42")
        );
    }
}
No SMTP server. No SendGrid account. No network. No side effects. Pure, fast, deterministic unit test. This is exactly what DIP enables — and why tech leads insist on it.
✅ Spring Wiring — How DI implements DIP in practice
@Configuration
public class NotificationConfig {

    // Spring injects the correct implementation automatically.
    // OrderService never knows which one it gets — nor should it.
    @Bean
    public NotificationSender notificationSender() {
        return new SendGridNotificationSender();
    }
}

    public void charge(double amount) {
        System.out.println("Charging " + amount + " via Stripe");
    }
}

public class PaymentProcessingService {
    private final MySqlPaymentRepository repository;
    private final StripePaymentGateway gateway;

    public PaymentProcessingService() {
        this.repository = new MySqlPaymentRepository();
        this.gateway    = new StripePaymentGateway();
    }

    public void processPayment(String orderId, double amount) {
        Payment payment = new Payment(orderId, amount, "PROCESSING");
        gateway.charge(amount);
        payment.setStatus("COMPLETED");
        repository.save(payment);
    }
}

public class Payment {
    private String id;
    private double amount;
    private String status;

    public Payment(String id, double amount, String status) {
        this.id = id; this.amount = amount; this.status = status;
    }
    public String getId()            { return id; }
    public double getAmount()        { return amount; }
    public String getStatus()        { return status; }
    public void setStatus(String s)  { this.status = s; }
}

1. What are the two rules of DIP, stated in your own words?

Rule 1: High-level modules (business logic, domain, use cases) must not depend on low-level modules (databases, APIs, file systems, messaging).

Rule 2: Both should depend on abstractions (interfaces). Abstractions must not depend on details — details must depend on abstractions.

2. What is the difference between DIP (the principle) and Dependency Injection (the pattern)?

DIP is a design principle about the direction of dependencies — it tells you what should depend on what. Dependency Injection is a pattern/technique — one way to achieve DIP by supplying dependencies from outside the class. You can violate DIP even while using DI if you inject a concrete class instead of an interface.

3. In Hexagonal Architecture (which you use at work), what are the "abstractions" and what are the "details" in DIP terms?

Abstractions = Ports — the interfaces defined in the domain layer (e.g. PaymentRepository, NotificationSender, EventPublisher).

Details = Adapters — the concrete implementations that handle infrastructure (e.g. MongoPaymentRepository, KafkaEventPublisher, SmtpNotificationSender). The domain defines the contract; infrastructure fulfils it.

4. Can you violate DIP even while using constructor injection? How?

Yes. If you write public OrderService(SmtpEmailSender sender) — injecting a concrete class through the constructor — you are using DI but violating DIP. The high-level module is still coupled to the low-level detail. DIP requires the parameter to be an interface: public OrderService(NotificationSender sender).

5. Why does DIP make unit testing easier? Give a concrete example.

When a high-level class depends on an interface, you can pass a mock in tests instead of a real implementation. Example: NotificationSender mockSender = Mockito.mock(NotificationSender.class);. This means OrderService can be tested without an SMTP server, network connection, or any infrastructure running. Without DIP (depending on the concrete class), Mockito cannot override the behaviour and you are forced to test against the real dependency.

6. Your team is adding Kafka as an event bus. How would you apply DIP when integrating it into a domain service?

Define an interface in the domain layer: public interface OrderEventPublisher { void publishOrderPlaced(Order order); }. Create a Kafka adapter that implements it: public class KafkaOrderEventPublisher implements OrderEventPublisher. The domain service receives OrderEventPublisher via constructor injection. It never imports any Kafka class. If you later replace Kafka with Azure Service Bus, only the adapter changes — zero impact on the domain.
